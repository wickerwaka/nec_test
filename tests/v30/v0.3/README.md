# V30 (uPD70116) SingleStepTests-format suite — tranche v0.3

CPU test data captured from a real NEC uPD70116C-8 on the nec_test
harness (DE10-Nano, max mode, 4 MHz, zero wait states), in the
SingleStepTests V20 schema extended for the 16-bit bus per the 8086-suite
precedent. Generated by `sw/emit_suite.py` (seed base `v30-v0.2`; every
case is reproducible from `sha-derived seed f"{base}/{opcode}/{n}"` —
rerolled seeds logged in `emit_log.txt`, and the output-index -> seed-attempt
map is recorded per form in `XX.seeds.json`).

This tranche is the **10,000-cases-each / 347-form** full-scale emission
(the v0.2 1,000-case suite extended in place: the first 1,000 cases of every
form are byte-identical to v0.2, same seed scheme, so v0.3 is a self-contained
superset). Forms: 332 opcode forms (`OPCODES`) + 15 pin-event forms — the same
347-form set that v0.2 validated. **String I/O (INS/OUTS, 6C-6F) is not in this
tranche** and is planned as a follow-up (the emitter's string-I/O generator and
the board-image port-serving for INS land after this run).

The suite was validated host-side before publication (`sw/validate_suite.py`):
schema/type over every record, SHA1 hash recompute, **independent final-RAM
reconstruction from the MEMW write transactions in `cycles`** (address from the
T1 address-phase row, data from the T2 data-phase row, UBE + address parity for
byte/word width), cold/prefetched convention, and trace open/close boundaries.

## Pin-event forms

15 pin-event (interrupt / HALT / POLL) forms at 10,000 cases each:
`INT.90 INT.B8 INT.8ED0 INT.8ED8 INT.FB INT.9D INT.F3AA` (maskable INT
against NOP / MOV imm / sreg-load shadows / EI / POP PSW / REP STM),
`NMI.90 NMI.B8` (edge NMI), `IE0.90` (masked INT), `POLL.LO POLL.REL`
(POLL pin low / released while waiting), `HLT.INT HLT.NMI HLT.RES`
(HALT wake by INT / NMI / masked-INT resume).

Pin-event schema extensions (see `docs/facts/interrupt_model.md`):

- `evt`: `{pin: 0|1|2 (INT/NMI/POLL), hold: N (0 = held to end),
  trigger: "fetch"|"fpop", addr: linear (fetch), delay: D}` — fetch
  mode asserts the pin during cycle `idx(CODE T1 at addr) + 2 + D`
  (the harness scheduler law); fpop mode during cycle
  `idx(window-opening F pop) + D`, `D >= 1` (used for prefetched
  variants, whose preload run-up does not exist in replay).
- `pins`: static pin levels before the event (bit2 = POLL_N high).
- `iord`: the 16-bit value the system returns for I/O reads (IN forms;
  default FFFF elsewhere).
- `close_addr`: vectored cases close their cycle window at the first
  F pop fetched from this (handler) address instead of a fixed pop
  count; the IVT entry points directly at the store location, so
  `final.cs:ip` is the handler entry.
- **`final.flags` for vectored (interrupt/trap) cases is the architectural value
  at interrupt entry = the interrupt-PUSHED PSW with IE/BRK cleared** (`pushed & ~0x300`),
  recoverable from the case's own `final.ram` (the PSW push at `SS:(dumped_sp+4)`; the
  harness store-stub dumps `sp = SP_at_int - 6`). It is NOT the post-handler store-stub
  PSW, which is an unreliable capture. This is the convention consumers should apply for
  pin-event/trap forms; the emitter derives it natively. (See
  `docs/notes/v02_suspected_divergences.md`.)
- Interrupt entry pushes appear in `final.ram` (PSW/PS/PC at SS:SP-2/
  -4/-6); `final.flags` has IE and BRK cleared.

## Files

`XX.json.gz` / `XX.N.json.gz` (modrm-group) / `0FXX.json.gz` — one JSON
array of test objects per opcode form. `XX.seeds.json` — per-form
output-index -> seed-attempt provenance map (`{seed_base, op, map:[[out_idx,
seed_i],...]}`). `metadata.json` — per-opcode status/flags-mask database
(V20 masks are valid for the V30: undefined-flag behavior is bit-exact,
see `docs/facts/undefined_flags.md`).

## Test object schema

Identical to the V20 suite (`docs/notes/singlesteptests_v20.md`):
`name`, `bytes`, `initial{regs,ram,queue}`, `final{regs,ram,queue}`,
`cycles`, `hash`, `idx`. Registers use Intel names (`ax..di, ip, flags`);
NEC aliases are listed in `metadata.json`. `final.regs` is sparse;
`final.ram` lists only changed bytes in access order. `final.ip` is the
architectural end-of-instruction value (the harness store-stub pad is
normalized out); for divide-trap cases it is the handler address
(IVT 0 -> 0000:0400, V20 convention — the 5-byte far-jump handler at
0x0400 is part of `initial.ram`).

Every odd-indexed case is **prefetched**: `initial.queue` holds the real
queue contents at the instruction's first-byte pop (6 bytes from even
addresses, 5 from odd — "as full as possible"), produced by a `63 C0`
preload run-up (verified side-effect-free on this silicon; the preload
bytes live below `cs:ip` and are not part of the record). Even-indexed
cases start from an empty queue. Consumers add `len(queue)` to the fetch
pointer as usual.

## Cycle rows (11 columns, 8086-suite shaped)

Columns 0, 3, and 4 are **synthesized** on the harness (there is no i8288
bus controller in the socketed-chip path); every other column is a direct
hardware sample. The synthesized columns are marked below.

| Idx | Field | Notes |
|---|---|---|
| 0 | pin bitfield | bit0 = ALE, **synthesized** as T1 (no i8288 on the harness); INTR/NMI bits always 0 |
| 1 | multiplexed bus (20-bit) | T1 rows: address-phase sample (valid address); other rows: PS3-0 status in bits 19:16 + data-phase AD15-0 |
| 2 | segment status | decoded from PS1:PS0, valid on T2/T3/Tw of active cycles, else `--` |
| 3 | memory command string (`RAW`) | **synthesized** from bus status + T-state (8288-equivalent) |
| 4 | IO command string (`RAW`) | **synthesized** likewise |
| 5 | UBE̅ (address phase) | 0 = upper byte enabled; even addr + 0 = 16-bit transfer, odd addr + 0 = high-byte transfer, even + 1 = low-byte |
| 6 | data bus (16-bit) | data-phase AD15-0 sample |
| 7 | bus status | decoded BS2-0 **address-phase sample**. NOTE: the V30 asserts the next cycle's status during T4, so T4 rows show the upcoming cycle's type (V20 traces show PASV there) |
| 8 | T-state | Ti/T1/T2/T3/Tw/T4 (harness FSM annotation) |
| 9 | queue op | QS1:QS0 point sample: F/S/E/- |
| 10 | queue byte | from the shadow-queue reconstruction (fetch data pushed at T4, low byte first on even word fetches) |

Trace window: from the row carrying the test instruction's first-byte
pop (`F`) to the row carrying the next instruction's `F` (inclusive) —
for trapping cases, the handler's first byte. `initial.ram` contains the
instruction bytes, operand/stack/IVT bytes, and every byte actually
fetched or read up to the window end (0x90 fill and, late in the stream,
harness store-stub bytes — recorded truthfully; consumers replaying
`initial.ram` reproduce the cycle rows exactly).

`hash` = SHA1 of
`json.dumps([name,bytes,initial,final,cycles],separators=(",",":"))`.

## Memory model — flat-1 MB validity

**Stated property: every case is verified flat-valid — the chip-captured behavior is
reproduced under a flat 1 MB memory model.** The capture harness's test RAM is 64 KB
mirrored across the 1 MB space (20-bit addresses alias to their low 16 bits), so a case
could in principle be shaped by a mirror-served byte a flat-memory emulator cannot
reproduce. Validity is established **behaviorally**, not by footprint: a case is
flat-valid iff replaying it on a flat 1 MB memory reproduces the captured `cycles` and
`final` state (verified by re-check). Mirror-dependent captures — where flat replay
diverges but 64 KB-mirrored replay matches — are **re-emitted** (confined per index)
until flat-valid. (Note: a mere mod-64K *footprint* alias is NOT a divergence — e.g. two
prefetch fetches into the 0x90-fill region alias to the same cell but both read the
default 0x90 on flat and mirror alike; only aliases that change an *observed* byte matter,
which is why the check is behavioral.) A chip-vs-emulator disagreement that is
memory-model-independent is retained as suite content, not rerolled away.

(Note: the upstream SingleStepTests **V20** suite does *not* guarantee flat-validity — a
small number of its cases are mirror-dependent; see `docs/notes/singlesteptests_v20.md`.)

## Known v0.3 limitations

- **No string I/O (INS/OUTS, 6C-6F)**: emitter string-I/O generator and
  the board-image per-case port-serving for INS are a follow-up tranche.
- **No IN/port-read randomization beyond the fixed forms** already present
  (E4/E5/EC/ED); per-case varied port data is part of the 6C-6F follow-up.
- ALE and the 8288 command columns are synthesized (documented above);
  bus-status T4 rows differ from V20 shape (hardware-truthful).
- Flags in initial states never have TF set; reserved PSW bits read as
  the V30 forces them (15:12=1, bit1=1, bits 5,3=0).

## Generation conditions

uPD70116C-8, max (large) mode, CPU clock 4 MHz, zero wait states,
socketed on the nec_test DE10-Nano harness (`hdl/`), state
injection/extraction per `docs/notes/loadstore_design.md`, persistent
serve runner (`sw/v30run.py`). Golden truth source is the socketed real
chip only (`EMIT_USE_CORE=False`; the internal `v30_core` is the DUT under
test, never the reference). Emitted 2026-07.
